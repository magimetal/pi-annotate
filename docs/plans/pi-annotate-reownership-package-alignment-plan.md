# Pi Annotate Reownership and Package Alignment Plan

## Objective
Realign `pi-annotate` to sibling-package standards used by `pi-system-prompt` and `pi-gizmo` without changing runtime behavior. Result must keep `/annotate` command and `annotate` tool behavior unchanged, preserve upstream/source provenance, credit original developer clearly, move maintained repository ownership/origin to `git@github.com:magimetal/pi-annotate.git`, and enforce explicit install-surface policy plus artifact-level verification before implementation can be considered done.

## Scope
- Non-functional repository/package realignment only
- Package layout, metadata, attribution, packaging, tests, and CI
- Install-surface policy and verification for git/local-path vs npm messaging
- No feature additions
- No behavior changes to annotation flow, native host protocol, browser extension UX, or output schema except metadata/documentation references required by repo move

## Out of Scope
- New annotator features
- Browser-extension UX redesign
- Native-host protocol changes
- Large code decomposition beyond what is needed for package layout and testability
- NPM ownership transfer mechanics beyond documenting and gating install instructions
- PRD/ADR creation unless hidden architecture scope appears during implementation

## Baseline Evidence
- **Observed:** package root currently exposes runtime entry files at `index.ts` and `types.ts` instead of sibling-style `extensions/` layout.
- **Observed:** `package.json` points repository, bugs, homepage, and gallery video URL to `nicobailon/pi-annotate`.
- **Observed:** `README.md` currently advertises `pi install npm:pi-annotate`, even though npm ownership/publish authority is not yet confirmed.
- **Observed:** `chrome-extension/popup.html` footer still links to upstream `nicobailon/pi-annotate#quick-start`.
- **Observed:** repo has no `tests/`, `tsconfig.json`, `vitest.config.ts`, or `.github/workflows/ci.yml`.
- **Observed:** `chrome-extension/native/host-wrapper.sh` is generated by `chrome-extension/native/install.sh` but also exists in repo and current dry-run tarball.
- **Observed:** version metadata is skewed: `package.json` is `0.4.2`, while `chrome-extension/manifest.json` and `chrome-extension/content.js` still advertise `0.4.0`.
- **Observed:** worktree is already dirty: `.gitignore`, `CHANGELOG.md`, `README.md`, `chrome-extension/content.js`, `chrome-extension/native/host-wrapper.sh`, `docs/`, `test-results/`.
- **Observed:** sibling package structure baseline exists in `../pi-system-prompt` (`extensions/`, `tests/`, `tsconfig.json`, `vitest.config.ts`, `.github/workflows/ci.yml`).
- **Observed:** attribution/provenance baseline exists in `../pi-gizmo` (`NOTICE.md`, README fork language, explicit maintained-separately wording).

## Target End State
- Explicit install-surface decision exists before public docs/metadata edits land.
- Until npm ownership is confirmed, public install guidance supports git and local package path only; npm instructions are removed or clearly marked unavailable.
- Runtime entry moves under `extensions/` with sibling-style package manifest paths.
- Package root remains installable by `pi install` from supported surfaces.
- Provenance and attribution become explicit in `README.md`, `NOTICE.md`, `CHANGELOG.md`, package metadata, and repo links.
- Packaging excludes generated artifacts and dev-only files while retaining required runtime/browser assets.
- Automated tests lock command/tool registration names plus tool schema shape and stable formatting surfaces.
- CI proves typecheck, tests, packed-artifact assertions, tarball-derived smoke install, and isolated project-scope `pi install -l` behavior.
- Repo remote/origin points to `git@github.com:magimetal/pi-annotate.git`.

## Install-Surface Policy Gate
- Until npm ownership/publish authority is confirmed, supported public install surfaces are maintained git source and local package path only.
- In unconfirmed state, `README.md`, `CHANGELOG.md`, `chrome-extension/popup.html`, and `package.json` metadata/examples must not advertise bare npm install instructions as available now.
- If npm ownership becomes confirmed during implementation, same change may restore npm install guidance only if all user-facing references are updated consistently and packed-artifact verification remains mandatory.
- Tarball verification is mandatory regardless of install-surface decision, because packed artifact is release boundary even when docs temporarily prefer git/local install.
